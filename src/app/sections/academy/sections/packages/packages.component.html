Einführung
Es gibt viele Webentwickler:innen da draußen und dementsprechend gibt es auch viele Zeilen Code, die von fast alle Entwickler:innen in irgendeiner Form in ihren Projekten verwendet werden. Diese Grundbausteine oder Bausätze stehen in Form von Paketen (Packages) zur freien Verfügung. Mit ihnen kannst Du komplexeren Code vereinfacht und mit nur wenigen Zeilen schreiben und das Package übernimmt die restliche Arbeit. Dies bringt einige Vorteile mit sich, hat aber auch ein paar Nachteile:
Du weißt nicht, wie sicher ein Package von einer fremden Person / Organisation ist. Wenn sie mit anderen Schnittstellen arbeiten, können Sicherheitslücken entstehen, die auf den ersten Blick nicht erkennbar sind.
Die Performance kann unter Packages leiden, je nach dem, wie gut sie optimiert sind.
Packages können schnell veralten und wenn sie nicht vom Anbieter des Pakets geupdated werden, können dort Fehler auftreten.
Wenn Du aber ungefähr weißt, wo und wie Du gute Pakete findest und sie von eher schlechteren unterscheiden kannst, bringen Pakete deutlich mehr Vorteile als Nachteile. In diesem Abschnitt zeigen wir Dir, wie Du Pakete evaluieren kannst.
Evaluation
Bei der Einbeziehung von Packages musst Du ein paar Dinge beachten, um die richtigen Pakete auszuwählen und die Performance Deiner Anwendung nicht unnötig zu belasten.
Beliebtheit auf npm - anhand der Anzahl der Downloads im letzten Monat oder der letzten Woche.
Release Datum - je aktueller desto besser.
Anzahl der Releases - je häufiger geupdated wird, desto besser.
Tests - man kann überprüfen, inwiefern Tests eingebaut sind. Dabei sind Pakete, die durch Tests geprüft sind, meistens weniger anfällig gegenüber Bugs (Fehlern) als Pakete ohne Test.
Anzahl der offenen bzw. ungelösten Issues - je weniger, desto besser.
Beliebtheit auf GitHub - je mehr Sterne, Forks und Watches, desto besser.
Anzahl der Contributors - je mehr Leute am Code arbeiten, desto mehr Fehler fallen auf.
Anzahl der Abhängigkeiten - je weniger Abhängigkeiten das Paket von kleineren Paketen hat, desto besser.
Am Beispiel von ng-bootstrap (siehe rechts) fällt auf, dass die Issues recht hoch ausfallen. Beim genaueren Betrachten der Issues bei GitHub siehst Du jedoch auch, dass schon knapp 2000 Issues behandelt wurden. Dementsprechend wird an dem Paket durchgängig weitergearbeitet und auf Probleme eingegangen. Also solltest Du bei solchen Fällen bei GitHub Issues genauer hinsehen, um einschätzen zu können, inwiefern die Entwickler:innen auf neue Probleme eingehen.
Aufgabe
Binde ein Paket Deiner Wahl in Dein erstelltes Angularprojekt ein. Gehe dafür auf npmjs und suche nach einem Paket, das Dein Projekt verbessern oder erweitern kann. Du kannst auch einen Tutor fragen, ob er Dir ein Paket für Deinen Kontext empfehlen kann.
Stay updated!
Bei einem Paket, das zum jetzigen Zeitpunkt gut gewählt ist, kann schon in ein paar Monaten die Entwicklung eingestellt werden. So etwas sollte stets überprüft werden. Das gilt auch für regelmäßige Updates des jeweiligen Pakets. Daher solltest Du regelmäßig Deine Pakete auf den aktuellsten Stand bringen und überprüfen, ob sie immer noch die richtige Wahl für Dein Projekt sind. Eine Möglichkeit, diese Überprüfung umzusetzen, ist die Nutzung entsprechender Dependency-Manager. Ein Beispiel hierfür ist depcheck.
