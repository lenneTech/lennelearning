Arbeiten als Team
Arbeiten
Wenn Du in einem Unternehmen arbeitest, bist Du im Normalfall Teil eines Teams. In jedem Projekt werden verschiedene Teams und Personen benötigt, die diverse Arbeiten am Projekt erledigen. Je nach Größe des Unternehmens kann auch die Größe und Zusammensetzung der Teams variieren. Daher gibt es verschiedene Aspekte im Hinblick auf den Ablauf und die Organisation eines gemeinsamen Projekts, die es zu beachten gilt. Dazu zählen die Arbeitsweise z. B. Scrum, die verwendeten Tools wie z. B. Jira und der vorgegebene Workflow. Diese Aspekte können sich je nach Unternehmen unterscheiden.
Methoden und Tools
Methoden und Tools
Für die Durchführung eines Projekts und die Zusammenarbeit nach den Vorgaben der Workflows haben Unternehmen meist bestimmte Tools und Methoden festgelegt, die verwendet werden sollen. Dabei sind vor allem agile Methoden wie z. B. Scrum bei Softwareprojekten sehr beliebt. Durch kurze Entwicklungszyklen, sogenannte Sprints, lassen sich Projekte flexibel steuern und anpassen. Ein Sprint dauert in der Regel ein bis vier Wochen. Für die Definition und Festlegung der Vorgänge, die innerhalb eines Sprints abgearbeitet werden sollen, können unterschiedliche Tools wie z. B. Jira verwendet werden. In den folgenden Videos bekommst Du einen kurzen Einblick in das Arbeiten mit Scrum und Jira.

Workflow
Meist existiert ein vordefinierter Workflow in einem Betrieb. Anhand eines Workflows wird einfach gesagt deutlich, wie verschiedene Personen und Teams zusammenarbeiten sollen. Ein Projekt umfasst in der Regel drei verschiedene Phasen:Konzeption, Design und Entwicklung (Umsetzung). Bedenke, dass nicht nur Entwickler:innen an einem Projekt arbeiten, sondern auch Personen aus anderen (nicht-technischen) Unternehmensbereichen. So haben bspw. Designer:innen vermutlich keine Programmierkenntnisse und den Entwickler:innen fehlt die Designexpertise. Genau deshalb ist es wichtig, entsprechend des Workflows zu agieren, sich um eine klare Kommunikation und gute Dokumentation zu bemühen z. B. durch die Projektseiten und Tickets in einem Projektboard sowie vordefinierte Regeln einzuhalten. Am verlinkten Beispiel schlüsseln wir exemplarisch einen Workflow auf.
Workflow
Im Backlog werden die Ideen und Anforderungen gesammelt. Diese werden sortiert und priorisiert.
In der Konzeption werden die Tickets für die Weiterbearbeitung konkretisiert. Abläufe, Datenbankmodelle und Co. werden spezifiziert.
Ist für ein Ticket ein Design nötig, werden im Schritt Design die UX/UI-Vorlagen erstellt. Diese werden mithilfe eines Clickdummies auf Vollständigkeit geprüft und von den Kund:innen abgenommen.
Im Schritt Development werden die Konzeption und die Designvorlage programmiertechnisch umgesetzt. Nicht zu vergessen sind hier Level 2 Tests. Außerdem werden Lanting-Regeln, Unit-Tests und e2e-Tests erstellt.
Im Enduser-Test findet der Level 3 Test statt.
Abhängig vom Ergebnis des Level 3 Tests landet das Ticket in der Review oder wird rückschrittig verlagert.
In der Review findet eine finale Abnahme des Tickets statt.
Alle freigegebenen Features landen in der Bereitstellung und werden Teil des nächsten Release sein.
Schnittstellen
Wird nach einem solchen Workflow gearbeitet, gibt es zwischen den Phasen immer Übergangsschritte. Zum Design geht es nur über den Schritt ToDesign, zur Review nur über den Schritt ToReview, etc. Diese Übergangsschritte sind die Schnittstellen zwischen den einzelnen Teams. Solche Schnittstellen sind in der Regel anfällig für Fehler, bergen aber auch häufig Potenzial für Prozessoptimierungen. Entsprechend sorgfältig sollte hier agiert werden.
So wird bspw. im Schritt ToDesign das vollumfängliche Ticket mit Beschreibungen, Skizzen, Beispielen sowie Hinweisen zu anderen Tickets für die weitere Verarbeitung als PDF hinterlegt. Tickets werden erst zum Design-Team weitergegeben, wenn diese freigegeben wurden.
Im Schritt ToDevelop liegen nun zusätzlich die abgenommenen Designvorlagen und der Clickdummy vor. Diese werden an die Entwickler übergeben, sodass diese direkt alle Designs und wichtigen Abläufe entnehmen können. Auch zu verwendende Bilder und Icons sollen direkt zur Verfügung stehen.
Wie genau diese Schritte definiert sind hängt immer davon ab, wie ein Workflow und dessen Regeln definiert sind. Es ist allerdings in jedem Fall wichtig, dass man sich an eben jenen hält, damit ein effizientes Arbeiten im Team möglich ist.
Git
Wenn man als Entwickler:in in einem Team arbeitet, bearbeitet man häufig gemeinsam den gleichen Code. Dabei können viele Probleme auftreten: Mehrere arbeiten am gleichen Problem und damit verliert man Zeit oder Du hast noch gar nicht den neuesten Stand des Projekts. Um diese Probleme zu vermeiden, nutzen wir Git und die damit kommenden Funktionen. In den nächsten Abschnitten erfährst Du unter anderem, wie Du Branches (Abzweigungen) des Hauptcodes erstellst, damit arbeitest und diese später wieder zum Hauptstrang hinzufügst.

Gästebuch
Setze die gelernten Inhalte in die Praxis um. Deine Aufgabe ist es, einen Eintrag in unserem Gästebuch zu hinterlassen. Wichtig ist, dass es hier prinzipiell verschiedene Methoden gibt, wie Du an und mit Repositories arbeiten kannst. Bist Du Teil eines Teams auf Github, so kannst Du in Repositories dieses Teams arbeiten. Bist Du das nicht, musst Du das Repository forken und dann anschließend eine Pull Request erstellen, mit welcher Dein Fork in das ursprüngliche Repository aufgenommen wird. In diesem Fall musst Du das Repository des Gästebuches erst forken.
Hier eine kurze Anleitung:
Forke das GitHub-Repository vom Gästebuch.
Bearbeite, was Du bearbeiten möchtest.
Navigiere wieder zum ursprünglichen Repository.
Über der Liste der Dateien wählst Du den Button Pull request aus.
Auf der Compare page vergleichst Du zwischen den Forks und prüfst, ob alles seine Richtigkeit hat.
Als base branch wählst Du das Repository aus, in welches Du Deinen Fork mergen möchtest.
Als head fork wählst Du Dein Repository aus, von dem Du mergen möchtest.
Füge einen aussagekräftigen Titel und Beschreibung hinzu.
Deine Pull request wird über den Button "Create Pull Request" finalisiert und Du stellst sie damit zur Review bereit.
Eine Review später siehst Du Deinen Fork gemerged im ursprünglichen Repository.
Github als Team
Wie gerade schon erwähnt gibt es auch die Möglichkeit, dass Du als Teil eines Teams Repositories bearbeitest. Sollte dies der Fall sein, folgt hier noch einmal die Anleitung dazu
Klone das GitHub-Repository des Projekts.
Erstelle einen neuen Branch, in welchem Deine Änderungen stattfinden werden.
Ändere, entwickle, reviewe, lösche oder was auch sonst anfällt.
Anschließend führst Du git add . aus. Danach committest Du mit git commit -m "hier kommt eine aussagekräftige Beschreibung rein".
Bevor Du Deine Änderungen pushst, gleichst Du nochmal Deinen Branch mit dem Hauptbranch ab. Dies machst Du um Mergekonflikte präventiv zu verhindern.
Dafür führst Du git checkout name-des-main-branches aus. Danach folgt git fetch und git pull.
Nun wechselst Du mit git checkout name-Deines-neuen-branches zurück in Deinen neu erstellten Branch und gleichst mit git rebase name-des-main-branches die zwei Stände miteinander ab.
Treten hier Konflikte auf, musst Du diese natürlich beheben.
Gibt es keine Konflikte, kannst Du nun mit git push Deine Änderungen auf Github als Pull Request veröffentlichen.
Auf der Seite des Repositories hast Du nun die Möglichkeit eine Pull Request zu erstellen. Diese sollte von einer anderen Person reviewed werden und wird dann, wenn alles in Ordnung ist, gemerged.
