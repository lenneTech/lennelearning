Arbeiten als Team
Arbeiten
Wenn Du in einem Unternehmen arbeitest oder arbeiten willst, bist Du im Normalfall Teil eines Teams. In jedem Projekt werden verschiedene Teams benötigt, die diverse Arbeiten am Projekt erledigen. Aus diesem Grund ist es wichtig, sich ein paar Dinge bewusst zu machen.
Workflow
Meist existiert ein vordefinierter Workflow in einem Betrieb. Anhand des gegebenen Workflows sieht man sehr deutlich wie verschiedene Teams miteinander arbeiten sollen. Hier ist es wichtig sich bewusst zu machen, dass nicht nur Entwickler:innen an einem Projekt arbeiten. Ein Projekt geht von der Konzeption übers Design bis zur Umsetzung. Das Team rund ums Design ist vermutlich nicht versiert mit dem Coding und das Team rund ums Coding nicht mit dem Designen. Genau deshalb ist es wichtig, dass man nach einem klaren Workflow agiert und sich an vordefinierte Regeln hält. Am verlinkten Beispiel schlüsseln wir exemplarisch einen Workflow auf.

Workflow
Im Backlog werden die Ideen und Anforderungen gesammelt. Diese werden sortiert und priorisiert.
In der Konzeption werden die Tickets für die Weiterbearbeitung konkretisiert. Abläufe, Datenbankmodelle und Co. werden spezifiziert.
Ist für ein Ticket ein Design nötig, werden im Schritt Design die UX/UI-Vorlagen erstellt. Diese werden mithilfe eines Clickdummies auf Vollständigkeit geprüft und vom Kunden abgenommen.
Im Schritt Development werden die Konzeption und die Designvorlage programmiertechnisch umgesetzt. Nicht zu vergessen sind hier Level 2 Tests. Außerdem werden Lanting-Regeln, Unit-Tests und e2e-Tests erstellt.
Im Enduser-Test findet der Level 3 Test statt.
Abhängig vom Ergebnis des Level 3 Tests landet das Ticket in der Review oder wird rückschrittig verlagert.
In der Review findet eine finale Abnahme des Tickets statt.
Alle freigegebenen Features landen in der Bereitstellung und werden Teil des nächsten Release sein.
Schnittstellen
Wenn man nach einem solchen Workflow arbeitet, befinden sich zwischen den Schritten immer Übergangsschritte. Zum Design geht es nur über den Schritt ToDesign, zur Review nur über den Schritt ToReview, etc. An solchen Übergangsschritten entstehen Schnittstellen zwischen den einzelnen Teams. Entsprechend liegen besonders an diesen Schritten viele Fehlerquellen und potentielle Prozessoptimierungen. Ergo ist hier besondere Sorgfalt geboten.
So wird beispielsweise im Schritt ToDesign das vollumfängliche Ticket mit Beschreibungen, Skizzen, Beispielen sowie Hinweisen zu anderen Tickets für die weitere Verarbeitung als PDF hinterlegt. Tickets werden erst zum Design-Team weitergegeben, wenn diese freigegeben wurden.
Im Schritt ToDevelop liegen nun zusätzlich die abgenommenen Designvorlagen und der Clickdummy vor. Diese werden an die Entwickler übergeben, sodass diese direkt alle Designs und wichtigen Abläufe entnehmen können. Auch zu verwendende Bilder und Icons sollen direkt zur Verfügung stehen.
Wie genau diese Schritte definiert sind hängt immer davon ab, wie ein Workflow und dessen Regeln definiert sind. Es ist allerdings in jedem Fall wichtig, dass man sich an eben jenen hält, damit ein effizientes Arbeiten im Team möglich ist.
Git
Wenn man als Entwickler:in in einem Team arbeitet, bearbeitet man häufig gemeinsam den gleichen Code. Dabei können viele Probleme auftreten: Mehrere arbeiten am gleichen Problem und damit verliert man Zeit oder Du hast noch gar nicht den neuesten Stand des Projekts. Um diese Probleme zu vermeiden, nutzen wir Git und die damit kommenden Funktionen. In den nächsten Abschnitten erfährst Du unter anderem, wie Du Branches (Abzweigungen) des Hauptcodes erstellst, damit arbeitest und diese später wieder zum Hauptstrang hinzufügst.

Gästebuch
Setze die gelernten Inhalte in die Praxis um. Deine Aufgabe ist es, einen Eintrag in unserem Gästebuch zu hinterlassen. Wichtig ist, dass es hier prinzipiell verschiedene Methoden gibt, wie Du an und mit Repositories arbeiten kannst. Bist Du Teil eines Teams auf Github, so kannst Du in Repositories dieses Teams arbeiten. Bist Du dies nicht, musst Du das Repository forken und dann anschließend eine Pull request erstellen, mit welcher dein Fork in das ursprüngliche Repository aufgenommen wird. In diesem Fall musst Du das Repository des Gästebuches erst forken.
Hier eine kurze Anleitung:
Forke das GitHub-Repository vom Gästebuch.
Bearbeite, was Du bearbeiten möchtest.
Navigiere wieder zum ursprünglichen Repository.
Über der Liste der Dateien wählst Du den Button Pull request aus.
Auf der Compare page vergleichst Du zwischen den Forks und prüfst, ob alles seine Richtigkeit hat.
Als base branch wählst Du das Repository aus, in welches Du deinen Fork mergen möchtest.
Als head fork wählst Du dein Repository aus, von dem Du mergen möchtest.
Füge einen aussagekräftigen Titel und Beschreibung hinzu.
Deine Pull request wird über den Button "Create Pull Request" finalisiert und Du stellst sie damit zur Review bereit.
Eine Review später siehst Du deinen Fork gemerged im ursprünglichen Repository.
Github als Team
Wie gerade schon erwähnt gibt es auch die Möglichkeit, dass Du als Teil eines Teams Repositories bearbeitest. Sollte dies der Fall sein, folgt hier noch einmal die Anleitung, wie Du in diesem Fall am Repository arbeitest
Klone das GitHub-Repository des Projekts.
Erstelle einen neuen Branch, in welchem deine Änderungen stattfinden werden.
Ändere, entwickle, reviewe, lösche oder was auch sonst anfällt.
Anschließend führst Du git add . aus. Danach committest Du mit git commit -m "hier kommt eine aussagekräftige Beschreibung rein".
Bevor Du deine Änderungen pusht, gleichst Du nochmal deinen Branch mit dem Hauptbranch ab. Dies machst Du um Mergekonflikte präventiv zu verhindern.
Dafür führst Du git checkout name-des-main-branches aus. Danach folgt git fetch und git pull.
Nun wechselst Du mit git checkout name-deines-neuen-branches zurück in deinen neu erstellten Branch und gleichst mit git rebase name-des-main-branches die zwei Stände miteinander ab.
Treten hier Konflikte auf, musst Du diese natürlich beheben.
Gibt es keine Konflikte, kannst Du nun mit git push deine Änderungen auf Github als Pull Request veröffentlichen.
Auf der Seite des Repositories hast Du nun die Möglichkeit eine Pull Request zu erstellen. Diese sollte von einer anderen Person reviewed werden und wird dann, wenn alles in Ordnung, ist gemerged.
