Arbeiten als Team
Arbeiten
Wenn du in einem Unternehmen arbeitest oder arbeiten willst, bist du im Normalfall Teil eines Teams. In jedem Projekt werden verschiedene Teams benötigt, die diverse Arbeiten am Projekt erledigen. Aus diesem Grund ist es wichtig, sich ein paar Dinge bewusst zu machen.
Workflow
Meist existiert ein vordefinierter Workflow in einem Betrieb. Anhand des gegebenen Workflows sieht man sehr deutlich wie verschiedene Teams miteinander arbeiten sollen. Hier ist es wichtig sich bewusst zu machen, dass nicht nur Entwickler an einem Projekt arbeiten. Ein Projekt geht von der Konzeption übers Design bis zur Umsetzung. Das Team rund ums Design ist vermutlich nicht versiert mit dem Coding und das Team rund ums Coding nicht mit dem Design. Genau deshalb ist es wichtig, dass man nach einem klaren Workflow agiert und sich an vordefinierte Regeln hält. Am verlinkten Beispiel schlüsseln wir exemplarisch einen Workflow auf.

Workflow
Im Backlog werden die Ideen und Anforderungen gesammelt. Diese werden sortiert und priorisiert.
In der Konzeption werden die Tickets für die Weiterbearbeitung konkretisiert. Abläufe, Datenbankmodelle und co werden spezifiziert.
Ist für ein Ticket ein Design nötig, werden im Schritt Design die UX/UI-Vorlagen erstellt. Diese werden mithilfe eines Clickdummies auf Vollständigkeit geprüft und vom Kunden abgenommen.
Im Schritt Development werden die Konzeption und die Design-Vorlage programmiertechnisch umgesetzt. Nicht zu vergessen sind hier Level 2 Tests. Außerdem werden Lanting-Regeln, Unit-Tests und e2e-Tests erstellt.
Im Enduser-Test findet der Level 3 Test statt.
Abhängig vom Ergebnis des Level 3 Tests landet das Ticket in der Review oder wird rückschrittig verlagert.
In der Review findet eine finale Abnahme des Tickets statt.
Alle freigebenden Features landen in der Bereitstellung und werden Teil des nächsten Release sein.
Schnittstellen
Wenn man nach einem solchen Workflow arbeitet befinden sich zwischen den Schritten immer Übergangsschritte. Zum Design geht es nur über den Schritt ToDesign, zur Review nur über den Schritt ToReview, etc. An solchen Übergangsschritten entstehen Schnittstellen zwischen den einzelnen Teams. Entsprechend liegen besonders an diesen Schritten viele Fehlerquellen und potentielle Prozessoptimierungen. Ergo ist hier besondere Sorgfalt geboten.
So wird beispielsweise im Schritt ToDesign das vollumfängliche Ticket mit Beschreibungen, Skizzen, Beispielen, Hinweisen zu anderen Tickets für die weitere Verarbeitung als PDF hinterlegt. Tickets werden erst zum Design-Team weitergegeben, wenn diese freigegeben wurden.
Im Schritt ToDevelop liegen nun zusätzlich die abgenommenen Designvorlagen und der Clickdummy vor. Diese werden an die Entwickler übergeben, sodass diese direkt alle Designs und wichtigen Abläufe entnehmen können. Auch zu verwenden Bilder und Icons sollen direkt zur Verfügung stehen.
Wie genau diese Schritte definiert sind hängt immer davon ab, wie ein Workflow und dessen Regeln definiert sind. Es ist allerdings in jedem Fall wichtig, dass man sich an eben jenen hält, damit ein effizientes Arbeiten im Team möglich ist.
Git
Wenn man als Entwickler in einem Team arbeitet, bearbeitet man häufig gemeinsam den gleichen Code. Dabei können viele Probleme auftreten: Mehrere arbeiten am gleichen Problem und damit verliert man Zeit oder du hast noch gar nicht den neuesten Stand des Projekts. Um diese Probleme zu vermeiden, nutzen wir Git und die damit kommenden Funktionen. In den nächsten Abschnitten erfährst du unter anderem, wie du Branches (Abzweigungen) des Hauptcodes erstellst, damit arbeitest und diese später wieder zum Hauptstrang hinzufügst.

Gästebuch
Setze die gelernten Inhalte in die Praxis um. Deine Aufgabe ist es, einen Eintrag in unserem Gästebuch zu hinterlassen. Hier eine kurze Anleitung:
Klone das GitHub-Repository vom Gästebuch.
Erstelle einen neuen Branch mit dem Namen student/[DeinName] .
Schreibe in das Gästebuch einen Eintrag mit deinem Namen.
Führe git add . aus.
Führe git commit -m "rezension von [DeinName]" aus.
Bevor du pushst, solltest du schauen, ob sich, während du gearbeitet hast, etwas an dem Hauptstrang geändert hat. Es könnte zum Beispiel sein, dass ein anderes Akademiemitglied gleichzeitig eine Rezension schreibt. Dafür machst du Folgendes:
Führe git checkout main aus, um in den Hauptzweig zu wechseln.
Führe git fetch aus, um zu prüfen, ob Daten, die nicht in deinem Ordner sind, auf GitHub liegen.
Führe git pull aus, um die Daten, die nicht in deinem Ordner sind, von GitHub herunterzuladen.
Danach gehst du mit git checkout student/[DeinName] in deinen Branch
Und checkst mit git rebase main ob deine Änderungen ohne Konflikte bestehen bleiben können.
Wenn hier Konflikte auftreten, solltest du diese beheben.
Danach mit git push das ganze in GitHub pushen.
Jetzt gehst du in GitHub auf das Repository, gehst in den Tab Pull requests und forderst ein Review von deinem Tutor an.
Wenn alles in Ordnung ist, bekommst du eine Bestätigungsmail, ansonsten eine Mail mit Änderungswünschen - das alles siehst du auch auf GitHub.
Wenn Änderungswünsche vorliegen, solltest du diese vollziehen und ein neues Review fordern.
Wenn alles bestätigt ist und gemerged wurde, bist du fertig.
